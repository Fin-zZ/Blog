# web安全

# XSS(Cross Site scripting)跨站脚本攻击

恶意的代码混入用户端处代码里，而浏览器无法区分，导致被执行。 由于是在用户端处，所以能够获取到用户的cookie等个人信息。

# 入侵渠道

那又是哪些渠道被注入恶意代码的呢

  1. 通过用户UGC（user generate content）:输入框等
  2. url
  3. 第三方链接
  4. post表单参数
  5. cookie（子域的混入）
  6. Referer

# XSS分类

  1. 反射型攻击 构造出恶意的url，诱骗用户点击后，服务器端会返回html给浏览器，同时包含有恶意代码的片段也被返回和 被浏览器解析，导致恶意代码最终被执行。 常见于通过url传递参数的场景，比如 search会带上参数， 链接跳转；服务器 返回数据时也会带上输入的参数。还有post提交表单时也有可能发生。

  2. 存储型攻击 恶意代码被发送到服务器端数据库，其他用户访问该资源时，此处内容返回给浏览器， 且代码每次被访问都会执行。 比如商品评论，论坛发帖，文章，用户对话私信；

反射型和存储型的区别： 反射型通常借助url、存储型借助数据库返回恶意代码。

  1. DOM XSS型 攻击者构建恶意url，用户打开后，浏览器打开并解析，由前端js执行url中的恶意代码。 与前两种区别在于这是由浏览器自身解析和执行，未经过服务器端，属于前端js自身漏洞。

# 防范

攻击主要分成两类

  • 用户侧输入了恶意代码并提交
  • 浏览器返回并解析执行了恶意代码
  1. 防止用户侧输入的内容带有恶意代码
    • 输入侧针对明确要求的内容，可进行限制，例如手机号 邮箱
  2. 防止浏览器执行解析恶意代码
    • 防止在HTML中出现恶意代码 由于存储型和反射型XSS攻击都是插入到HTML里的， 所以需要一定防范。
      • 改为纯前端渲染，api只请求数据，使得html结构不会由接口返回定义
      • 如果不能纯前端渲染，则需对后台传来的html进行充分转译 第一种指的是，浏览器最开始加载一个无数据的html，之后按序加载JS标签并发起ajax请求， 接口拿到数据后，再去渲染页面，渲染的时候，各个数据也很明确属于哪个部分，比如 innerHtml，style等。 缺点时无法进行seo，且需要注意dom xss攻击 第二种 拼接html的，可以借助成熟的模版库协助。
    • 防止在JS中出现恶意代码 本质是js代码里不严谨，执行了不可信的js代码。 Vue/React针对v-html、dangerouslySetInnerHTML要谨慎 原生js，包括jquery需要注意innerHtml,outHtml,document.write()等， 包括js的eval函数

# 额外的防范

  1. 开启CSP(Content-Security-Policy),可以设置哪些域名下的资源可被加载。
  2. 开启http-only，这项是针对cookie只能被服务端读取，js无法获取到
